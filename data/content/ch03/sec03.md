## データレイク（Data Lake）

**定義**：
[[data-lake]]は、生データを元形式に近いまま蓄積し、後段の用途に応じて変換する前提のデータ基盤である。構造化・半構造化・非構造化データを同一基盤で管理でき、分析、機械学習、再処理の共通ソースとして機能する。

**特徴**：
- [[schema-on-read]]で、利用時にデータ構造を解釈するため柔軟性が高い。
- オブジェクトストレージ中心でコスト効率が良く、大容量保持に向く。
- 多様なユースケースへ展開しやすい一方、メタデータ管理が弱いと[[data-swamp]]化しやすい。
- 品質基準、命名規約、ライフサイクル管理を設計しないと再利用性が低下する。

**技術**：
- Amazon S3
- Azure Data Lake Storage
- Google Cloud Storage
- HDFS（Hadoop Distributed File System）

実務では、[[data-catalog]]、アクセス制御、データ品質監視を併設し、単なる保存庫ではなく「検索可能で信頼できるレイク」にすることが重要である。

## データウェアハウス（Data Warehouse）

**定義**：
[[data-warehouse]]は、分析用途に最適化された構造化データ基盤であり、[[bi]]の標準レポートや経営指標集計を安定運用するために設計される。データモデルを事前定義し、再現可能な集計と高い説明責任を実現する。

**特徴**：
- [[schema-on-write]]で取り込み時に整形・検証するため、利用時の一貫性が高い。
- 列指向処理や最適化機構により、集計クエリ性能を出しやすい。
- 品質統制と監査対応に向くが、要件変化にはモデル改修計画が必要。
- 取り込み前の変換品質が低いと、誤った指標が全社へ展開されるリスクがある。

**技術**：
- Snowflake
- BigQuery
- Redshift
- Azure Synapse Analytics

データウェアハウスは「正しさ」と「定常運用」を重視するため、KPI管理、定例報告、監査証跡が重要な領域で特に有効となる。

## データレイクハウス（Data Lakehouse）

**定義**：
[[data-lakehouse]]は、[[data-lake]]の柔軟性と[[data-warehouse]]の整合性・分析性能を統合するアーキテクチャである。低コストストレージ上に、トランザクション管理やメタデータ統制を重ねることで、BIとMLを同一基盤で扱いやすくする。

**特徴**：
- オブジェクトストレージを活用しつつ、[[acid]]相当の更新整合を担保できる。
- スキーマ強制、タイムトラベル、バージョン管理により再現性を高める。
- バッチとストリーミングを統合しやすく、分析と機械学習の往復がしやすい。
- ただし、フォーマット選定と運用ルールを誤ると性能や互換性で課題が出る。

**テーブルフォーマット**：
- [[delta-lake]]：トランザクションログに基づく更新管理とタイムトラベルを提供。
- [[apache-iceberg]]：大規模テーブル管理、スキーマ進化、パーティション進化に強い。
- [[apache-hudi]]：増分更新とストリーミング連携を重視した設計。

選定基準は、更新パターン（追記中心か更新中心か）、エンジン互換性、運用体制、既存ツール連携で判断するのが実務的である。

## メダリオンアーキテクチャ（Medallion Architecture）

[[medallion-architecture]]は、データ品質と利用目的に応じてデータを段階管理する運用モデルである。レイク/レイクハウスで、再現性と説明可能性を担保しながら段階的に価値を高めるために使われる。

**ブロンズ層（Raw）**：
- 取得元データをできるだけ元の形で保持し、監査と再処理の起点にする。
- 欠損や異常を含みうるため、直接業務指標には使わない。

**シルバー層（Cleaned）**：
- クレンジング、型変換、重複排除、参照整合を実施し、分析可能状態へ整える。
- 共通ディメンションやキー統一を行い、部門横断利用を可能にする。

**ゴールド層（Aggregated）**：
- KPI算出や業務ロジックを適用した提供データを配置する。
- BIダッシュボード、定例レポート、特徴量配信など利用目的別に最終形を提供する。
- 変更管理を徹底し、下流への影響範囲を把握できる形で運用する。

試験では、各層の責務混同が頻出である。ブロンズで集計済みデータを作る、シルバーを飛ばしてゴールドへ直結する、などの設計は品質・再現性の観点で不適切と判断される。

▼覚えておくポイント
- [[data-lake]]は柔軟性、[[data-warehouse]]は整合性と定常分析運用が強み。
- [[data-lakehouse]]は統合基盤として有効だが、フォーマットと運用設計が成功条件。
- [[delta-lake]]、[[apache-iceberg]]、[[apache-hudi]]は更新特性と互換性で選ぶ。
- [[medallion-architecture]]は品質段階管理で再現性と再利用性を高める。
