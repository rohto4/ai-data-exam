## 主要クラウドプロバイダーのデータサービス

### AWS（Amazon Web Services）

**ストレージ**：
- [[amazon-s3]]：データレイク基盤として使われるオブジェクトストレージ。
- EBS：仮想マシン向けブロックストレージ。
- FSx：高性能ファイルワークロード向けマネージドファイルストレージ。

**データベース**：
- RDS：マネージドRDBMSで運用負荷を下げやすい。
- [[amazon-dynamodb]]：低遅延アクセスに強いマネージドNoSQL。
- [[amazon-redshift]]：分析特化型データウェアハウス。
- Keyspaces：Cassandra互換のマネージドサービス。

**分析・ETL**：
- [[aws-glue]]：ETL/カタログ統合サービス。
- [[amazon-athena]]：S3上データへサーバーレスSQL実行。
- EMR：Hadoop/Spark実行基盤をマネージド提供。
- [[amazon-kinesis]]：ストリーミング収集・処理基盤。

AWSはサービス選択肢が広く、細かな要件に合わせやすい一方、設計自由度が高いため標準化を怠ると構成が複雑化しやすい。

### Google Cloud Platform（GCP）

**分析基盤**：
- [[bigquery]]：サーバーレス分析基盤で、運用負荷を抑えつつ大規模集計を行いやすい。
- [[google-cloud-dataflow]]：バッチ/ストリーミング統合処理。
- Dataproc：Spark/Hadoopを迅速に実行できるマネージド環境。
- [[google-pubsub]]：イベント駆動連携向けメッセージング。

**データベース**：
- Cloud SQL：マネージドRDBMS。
- Firestore：アプリケーション連携しやすいドキュメントDB。
- Bigtable：大規模時系列・低遅延用途に強いワイドカラムDB。
- [[cloud-spanner]]：グローバル分散とトランザクション整合を両立するDB。

GCPは分析・機械学習系サービスとの親和性が高く、データ基盤をサーバーレス中心で組みやすい点が強みである。

### Microsoft Azure

**分析基盤**：
- [[azure-synapse-analytics]]：DWHと分析を統合した基盤。
- [[azure-data-factory]]：ETL/ELTパイプライン構築サービス。
- Stream Analytics：リアルタイム集計サービス。
- Azure Databricks：Spark実行・分析連携基盤。

**データベース**：
- Azure SQL Database：マネージドSQL Server。
- [[azure-cosmos-db]]：グローバル分散NoSQL。
- PostgreSQL/MySQL：マネージドOSS系RDBMS。

AzureはMicrosoft製品との統合や企業IT統制との整合を取りやすく、既存AD基盤や業務システム連携が重視される環境で採用しやすい。

## マルチクラウド戦略

**メリット**：
- [[vendor-lock-in]]の緩和。
- 強みの異なるサービスを組み合わせるベストオブブリード戦略。
- 障害分散による事業継続性向上。
- 価格・性能比較による調達柔軟性の向上。

**課題**：
- 運用標準、監視、障害対応の複雑化。
- チームに複数クラウド知識が必要となる。
- クラウド間データ転送コストと遅延が発生しやすい。
- ID管理・鍵管理・監査統制が分散しやすい。

**実装パターン**：

**クラウドバースト**：
- 通常時は既存基盤、ピーク時のみ外部クラウドを活用する方式。
- キャパシティ最適化に有効だが、接続設計とデータ同期が前提となる。

**災害復旧（DR）**：
- プライマリ障害時に別クラウドへ切替える設計。
- 復旧手順の定期訓練とデータ整合確認が不可欠。

**機能別分散**：
- 分析基盤と業務基盤をクラウドごとに分離して最適化する方式。
- データ連携仕様を標準化しないと保守負荷が増える。

マルチクラウドは「導入すること」自体が目的ではなく、可用性・機能最適化・調達リスク分散など明確な目的に紐づけて採用判断するのが重要である。

## クラウド選定の基準

| 観点 | 評価ポイント |
|------|-------------|
| パフォーマンス | レイテンシ、スループット |
| コスト | 従量課金、リザーブドインスタンス |
| 機能 | 必要なサービスの有無 |
| エコシステム | 既存システムとの親和性 |
| サポート | SL A、技術サポート |

選定実務では、以下の追加観点を整理すると判断が安定する。
- データ主権・法令対応（保存リージョン、監査証跡）。
- 運用モデル（IaC、監視、権限統制）との適合。
- 将来移行性（データ形式、API互換性、契約条件）。
- 人材と学習コスト（既存スキルとの連続性）。

試験では、単純な機能比較よりも「要件に対して何を優先して選ぶか」を問う設問が多い。例えば、分析速度優先か、統制優先か、既存連携優先かで最適解は変わる。

▼覚えておくポイント
- AWS/GCP/Azureは強みが異なるため、機能・運用・統制の観点で適材適所を判断する。
- [[multicloud]]は有効だが、運用標準と統合監視を先に設計しないと複雑化する。
- クラウド選定はコストだけでなく、性能、法令対応、将来移行性を含めて総合評価する。
